# План добавления недостающих возможностей Twig 3.x в парсер tree-sitter-twig

## Текущее состояние

Парсер tree-sitter-twig уже поддерживает большую часть синтаксиса Twig 3.x, включая:

- Все основные теги (`if`, `for`, `block`, `extends`, `include`, `embed`, `macro`, `import`, `from`, `use`, `set`, `with`, `autoescape`, `sandbox`, `verbatim`, `apply`, `cache`, `deprecated`, `do`, `flush`)
- Выражения: бинарные операторы (включая `and`, `or`, `not`, `b-and`, `b-or`, `b-xor`, `==`, `!=`, `<=>`, `<`, `>`, `<=`, `>=`, `in`, `not in`, `matches`, `starts with`, `ends with`, `has some`, `has every`, `..`, `+`, `-`, `*`, `/`, `//`, `%`, `~`, `**`, `??`, `is`, `is not`), унарные операторы (`+`, `-`, `not`), тернарный оператор (`?:`), фильтры, вызовы функций, доступ к свойствам, элементы массива, объекты, стрелочные функции, интерполяция строк, spread оператор, slice.
- Комментарии и inline-комментарии.
- Управление пробелами (`-`, `~`).

## Обнаруженные недостающие возможности

После изучения грамматики (`src/grammar.json`), тестов (`corpus/`) и документации Twig 3.x выявлены следующие пробелы:

### 1. Тесты (tests)

Twig предоставляет набор **тестов**, которые используются с оператором `is`:

- `defined` – проверка определённости переменной.
- `empty` – проверка на пустоту.
- `even` – чётность числа.
- `odd` – нечётность числа.
- `iterable` – является ли значение итерируемым.
- `constant` – проверка, является ли значение константой.
- `null` – проверка на `null` (уже может работать как `is null`, но стоит убедиться).
- `same as` – уже поддерживается.
- `divisible by` – уже поддерживается.

**Текущая реализация:** В грамматике тесты `divisible by` и `same as` представлены как отдельные операторы, но остальные тесты отсутствуют. Вероятно, они должны быть интегрированы в правило `binary_expression` с оператором `is`/`is not` и правой частью – **test_expression**.

### 2. Макросы со значениями по умолчанию для аргументов

Синтаксис макросов позволяет указывать значения по умолчанию:

```twig
{% macro input(name, value = 'default') %}
```

В текущей грамматике аргументы макроса используют то же правило `arguments`, что и вызовы функций, где допустимы только позиционные и именованные аргументы (`key: value`). Однако синтаксис `identifier = expression` для значений по умолчанию не поддерживается.

### 3. Цикл `for` с условием `if` (устаревший)

Устаревшая, но ещё работающая возможность:

```twig
{% for item in items if item.active %}
```

В грамматике отсутствует опциональная часть `if expression` после `in expression`.

### 4. Операторы идентичности `===` и `!==`

Twig 3.x поддерживает строгое сравнение с учётом типов:

```twig
{{ a === b }}
{{ a !== b }}
```

В грамматике присутствуют `==` и `!=`, но `===` и `!==` отсутствуют.

### 5. Тег `trans`

Тег для переводов (интернационализация):

```twig
{% trans %}Hello{% endtrans %}
```

Сейчас парсер воспринимает `trans` как обычный тег (узел `tag`), что достаточно для синтаксического разбора, но не даёт семантической информации. Для улучшения подсветки и анализа можно выделить его в отдельный узел.

### 6. Устаревшие теги `filter` и `spaceless`

- `filter` – устарел, рекомендуется использовать `apply`.
- `spaceless` – устарел, удаляет пробелы между HTML-тегами.

Их поддержка не критична, но может быть добавлена для полноты.

### 7. Расширенный тег `deprecated`

Поддержка второго аргумента (версии):

```twig
{% deprecated 'Message' '1.2.3' %}
```

Сейчас принимается только одно выражение.

### 8. Тег `cache` с тегами (возможно, расширение Symfony)

Не является частью ядра Twig, но иногда используется. Требует уточнения.

## План реализации

### Этап 1: Подготовка (1 день)

1. Создать ветку для разработки.
2. Настроить окружение для тестирования (`npm run test`).
3. Изучить текущую структуру грамматики и тестов, чтобы понять, как добавлены `divisible by` и `same as`.

### Этап 2: Добавление тестов (2–3 дня)

1. В `src/grammar.json`:
   - Добавить правило `test_expression` как подкатегорию `expression` (или расширить `binary_expression`).
   - Определить список тестов как ключевые слова, которые могут следовать за `is`/`is not`.
   - Учесть, что некоторые тесты могут иметь аргументы в скобках (`divisible by(3)`), а другие – нет (`defined`).
2. Обновить `src/node-types.json`:
   - Добавить типы узлов для новых тестов (или обобщённый узел `test_expression` с полем `name`).
3. При необходимости обновить `src/scanner.c` для распознавания новых ключевых слов.
4. Добавить тесты в `corpus/tests.txt` (или новый файл `corpus/tests_additional.txt`).
5. Проверить, что все существующие тесты проходят.

### Этап 3: Операторы `===` и `!==` (0.5 дня)

1. В `src/grammar.json` добавить два новых оператора в `binary_expression` с тем же приоритетом, что и `==`/`!=`.
2. Обновить `src/node-types.json` (если требуется).
3. Добавить тесты в `corpus/expressions.txt`.

### Этап 4: Макросы со значениями по умолчанию (1–2 дня)

1. Исследовать, как Twig трактует синтаксис `identifier = expression` внутри `arguments`.
2. Создать отдельное правило `macro_arguments` или расширить существующее `arguments` поддержкой `default_argument`.
3. Внести изменения в грамматику и node-types.
4. Добавить тесты в `corpus/statements.txt`.

### Этап 5: Дополнительные теги (по желанию) (1 день)

1. Добавить тег `trans` как отдельный узел (опционально).
2. Добавить устаревшие теги `filter` и `spaceless` (можно реализовать как псевдонимы для `apply` и отдельный тег).
3. Расширить `deprecated` вторым аргументом.
4. Добавить соответствующие тесты.

### Этап 6: Тестирование и фиксация (1 день)

1. Запустить все тесты (`npm run test`).
2. Проверить, что парсер корректно работает на реальных шаблонах (можно использовать примеры из документации Twig).
3. Собрать WASM-версию (`npm run build-wasm`) и убедиться в отсутствии регрессий.
4. Обновить CHANGELOG.md (если требуется).
5. Создать pull request.

## Риски и ограничения

- **Конфликты грамматики**: Добавление новых правил может привести к конфликтам разрешения (уже есть секция `conflicts`). Необходимо проверить и при необходимости обновить её.
- **Совместимость с биндингами**: Изменения в node-types.json могут затронуть биндинги для Go, Rust и Node.js. Нужно убедиться, что изменения обратно совместимы (добавление новых типов узлов безопасно).
- **Производительность**: Увеличение грамматики может незначительно замедлить парсинг, но это маловероятно.

## Заключение

Реализация описанных возможностей значительно повысит покрытие Twig 3.x и улучшит полезность парсера для инструментов, использующих tree-sitter (подсветка, автодополнение, линтинг). Рекомендуется начать с тестов, так как они наиболее востребованы, а затем перейти к остальным пунктам в порядке приоритета.

План может корректироваться по мере обнаружения новых деталей.
